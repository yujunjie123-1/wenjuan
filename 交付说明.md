# survey-local-demo 交付说明

## 接手速览

项目根目录：

```text
E:\问卷软件制作\survey-local-demo
```

启动命令：

```powershell

.\gradlew.bat run
```

本地网页入口：

```text

```

本地样例问卷：

```text
http://localhost:8080/sample-form.html
```

一句话概括：这是一个 Kotlin + Ktor + Playwright Java 的本地问卷填报演示工具，前端上传 Excel 并配置映射，后端逐行打开可见浏览器填写问卷，可选真实点击提交。

## 技术栈

- Kotlin 1.9.25
- JVM toolchain 21
- Ktor 2.3.12
- Apache POI 5.2.5
- Playwright Java 1.48.0
- Gradle Wrapper
- 原生 HTML/CSS/JavaScript 前端

## 功能范围

- Excel 上传、保存和前 10 行预览。
- Excel 列与问卷题目字段映射。
- 支持填空题、单选题、多选题、下拉题、矩阵/量表等问卷星常见题型。
- 支持题号偏移，用于按问卷星题号直接定位。
- 支持按行范围执行、设置行间隔、设置每行填写总耗时。
- 支持桌面端、移动端、微信端 UA 来源比例。
- 支持代理池配置和按行代理状态记录。
- 支持可见浏览器执行，便于人工观察。
- 默认只填写并截图，可开启真实点击提交。
- 启用提交后每行不自动重试，避免重复提交。
- 记录任务状态、成功失败计数、日志、原始行数据、截图路径。
- 提交后兼容普通感谢页和问卷星抽奖完成页。
- 一键高信效度填充：无需 Excel，直接按问卷结构生成单选题与量表题答案，目标是让量表题保持较高内部一致性。

## 一键高信效度填充

入口：

- 前端按钮：`index.html` 中 `#highAlphaButton`。
- 前端函数：`app.js` 中 `startHighAlphaFill`。
- 后端接口：`POST /api/high-alpha/tasks`。
- 请求模型：`HighAlphaTaskRequest`。
- 执行器：`OneClickHighAlphaFiller.kt`。
- 命令行入口：`OneClickHighAlphaRunner.kt`。

执行逻辑：

- 每份问卷创建一个独立 browser context。
- 首次打开问卷页使用 30 秒导航超时，最多重试 2 次。
- 优先按问卷星题块识别：`.field.ui-field-contain[topic]` 和 `div[topic][type]`。
- `type='3'` 识别为单选题。
- `type='5'` 识别为单题量表题。
- `type='6'` 按矩阵量表处理，会尝试拆分矩阵行。
- 非问卷星页面才回退到通用 `.field/.question/...` 识别。
- 量表题按潜变量分块生成答案，同一组题围绕同一个 latent score 波动，以提高内部一致性。
- 单选题按选项数做轮转分布，避免所有样本固定同一选项。
- 第一道“是否/是/否/是否使用”筛选题优先选积极选项，避免被筛出。
- 提交后等待普通完成页或问卷星抽奖完成页，未检测到完成页只记 warn，不直接卡死。

本次针对目标问卷：

```text
https://v.wjx.cn/vm/Y0EQA2i.aspx
```

已确认页面源码中共有 7 个 `type='3'` 单选题和 18 个 `type='5'` 量表题。此前识别不到量表题的原因是量表题选项不是 `label` 或 `input[type=radio]`，而是 `.scale-rating a[val]` 结构；旧逻辑只按 label/radio 计数，因此 `type='5'` 题块的可点击选项数被算成 0。当前修复后，`clickableOptionCount` 与 `selectNthOption` 都支持 `a[val]`、`.scale-rating a`、`.scale-div a`。

## 后端模块职责

- `Application.kt`：Ktor 入口，安装 JSON、CORS、异常处理，注册静态资源和 API 路由。
- `AppPaths.kt`：集中定义项目运行目录，包括 `.survey-local-demo`、`uploads`、`screenshots`、`config`。
- `Models.kt`：定义前后端通信模型，包括 `WorkbookPreview`、`FieldMapping`、`StartTaskRequest`、`TaskSnapshot`。
- `ExcelService.kt`：负责 Excel 保存、预览、读取行数据、自动生成映射、把 IP 写回 Excel。
- `TaskStore.kt`：负责内存中的任务状态、日志追加、取消、成功失败计数。
- `QuestionnaireRunner.kt`：核心执行器，负责打开浏览器、创建上下文、逐行填写、翻页、验证码处理、提交、截图、记录 IP。
- `automation/browser/ContextInitializer.kt`：负责浏览器上下文创建、桌面端/移动端/微信端 UA 和设备指纹应用。
- `automation/browser/BrowserContextOptionsFactory.kt`：把自动化配置转成 Playwright browser context options。
- `automation/behavior/HumanBehaviorSimulator.kt`：模拟延迟、滚动、点击和输入节奏。
- `automation/captcha/CaptchaSolver.kt`：验证码处理入口，目前按配置创建 solver。
- `automation/config/AutomationConfigService.kt`：读取并解析 `config/automation-profiles.json`。
- `automation/proxy/ProxyPool.kt`、`ProxySessionManager.kt`：代理池读取、按行代理选择、成功失败反馈。
- `automation/fingerprint/InitScriptFactory.kt`：注入浏览器环境脚本。

## 前端模块职责

- `index.html`：本地管理页面结构，包含上传 Excel、配置映射、启动任务和日志展示区域。
- `app.js`：前端主逻辑，负责调用 API、渲染 Excel 预览、维护映射表单、启动任务、轮询任务状态、展示截图链接。
- `styles.css`：本地管理页面样式。
- `sample-form.html`：本地样例问卷，用于不依赖外部问卷时验证填写流程。

## API 路由

- `GET /api/health`：返回服务状态和运行数据目录。
- `GET /api/automation/profiles`：读取自动化配置文件。
- `POST /api/workbooks`：上传 Excel，保存到运行目录，并返回预览和 workbookId。
- `POST /api/tasks`：创建并启动填报任务。
- `GET /api/tasks/{id}`：查询任务状态、日志、截图记录和计数。
- `POST /api/tasks/{id}/cancel`：取消任务。
- `GET /api/artifacts/{taskId}/{fileName}`：读取任务截图 PNG。
- `GET /api`：返回 `survey-local-demo` 文本，用于简单识别服务。

## 数据模型要点

- `FieldMapping.excelColumn`：单个 Excel 列。
- `FieldMapping.excelColumns`：多个 Excel 列，用于多选二进制列等场景。
- `FieldMapping.questionTitle`：问卷题目文本，也可配合题号偏移定位。
- `FieldMapping.questionType`：`AUTO`、`TEXT`、`SINGLE_CHOICE`、`MULTIPLE_CHOICE`、`SCALE`、`SELECT`。
- `FieldMapping.valueMode`：`TEXT`、`ORDINAL`、`MULTI_BINARY_COLUMNS`。
- `FieldMapping.required`：空值时是否必须填写。
- `FieldMapping.offset`：问卷星题号偏移，大于 0 时优先按题号定位。
- `StartTaskRequest.submitEnabled`：是否点击提交按钮。
- `StartTaskRequest.changeIp`：是否启用代理换 IP。
- `StartTaskRequest.sourceRatioMobile`、`sourceRatioLink`、`sourceRatioWechat`：三类来源比例。
- `TaskSnapshot.logs`：任务日志数组，日志里可带 `rowNumber`、`screenshotPath`、`screenshotUrl`、`rowData`。

## 任务执行链路

1. 前端上传 Excel 到 `POST /api/workbooks`。
2. 后端 `ExcelService.saveAndPreview` 保存文件并返回 `WorkbookPreview`。
3. 前端配置映射后调用 `POST /api/tasks`。
4. 后端校验 `StartTaskRequest`，读取总行数，创建 `TaskSnapshot`。
5. `QuestionnaireRunner.start` 在 IO coroutine 中异步执行任务。
6. `QuestionnaireRunner.runTask` 解析自动化配置，计算行范围和来源比例。
7. 每行创建独立 browser context 和 page。
8. `fillRow` 打开问卷 URL，等待 DOM ready，短暂等待 network idle。
9. 按映射逐题调用 `fillQuestion`，需要翻页时点击下一页。
10. 等待达到目标填写时长，保存 `row-N-filled.png`。
11. 若 `submitEnabled=false`，记录 review artifact，进入下一行。
12. 若 `submitEnabled=true`，先尝试验证码处理，再点击提交。
13. 提交后 `waitForSubmitCompletion` 等待普通完成页或抽奖完成页。
14. 保存 `row-N-submitted.png`，记录提交截图。
15. 尝试记录当前 IP 并写回 Excel。
16. 行结束时将 page 跳到 `about:blank`，由外层 context 释放资源。
17. 所有行结束后任务状态变为 `COMPLETED`，被取消则变为 `CANCELLED`，异常则变为 `FAILED`。

## 本次修复记录

问题现象：日志显示某行已经 `fillRow completed successfully`，也出现 `Row N filled and submit button clicked.` 和截图记录，但浏览器停在 `v.wjx.cn/wjx/join/completemobile2...` 抽奖页，后续行不继续。

定位结论：问卷星开启抽奖/激励后，提交成功页不是普通感谢页，而是移动端抽奖完成页。页面包含“您的答卷已经提交”“恭喜您获得了 1 次抽奖机会”“立即抽奖”等文本。原处理没有把这类页面作为提交完成后的稳定状态，同时 `page.close()` 在该页面上存在卡住风险。

修复位置：

- `QuestionnaireRunner.kt` 的 submit 分支：点击提交后调用 `waitForSubmitCompletion`。
- `QuestionnaireRunner.kt` 的 `waitForSubmitCompletion`：识别普通完成页和抽奖完成页。
- `QuestionnaireRunner.kt` 的截图逻辑：提交前截图为 `filled`，提交后截图为 `submitted`。
- `QuestionnaireRunner.kt` 的 finally 逻辑：不再直接 `page.close()`，改为 `detachPageBeforeContextClose` 跳转 `about:blank`，再交给 context 释放。
- `QuestionnaireRunner.kt` 的 `saveScreenshot`：截图前等待页面稳定，full-page 截图失败后降级为 viewport 截图，两次失败则写入有效 PNG 占位图并记录 warn，避免截图失败导致整行失败。
- `QuestionnaireRunner.kt` 的问卷导航逻辑：首次进入问卷页的 `page.navigate` 显式超时提高到 30 秒，并增加 2 次尝试；部分代理首次加载问卷星较慢时，不再因为原 15 秒导航超时直接失败。

完成页识别规则：

- URL 包含 `completemobile`。
- URL 包含 `/wjx/join/complete`。
- URL 包含 `/join/complete`。
- 页面文本包含“您的答卷已经提交”。
- 页面文本包含“感谢您的参与”。
- 页面文本包含“恭喜您获得了”。
- 页面文本包含“立即抽奖”。
- 页面文本包含“抽奖机会”。

等待策略：

- 最长等待 12 秒。
- 每 500 毫秒轮询一次。
- 未检测到完成信号时记录 warn，再等待 2 秒 fallback settle，然后继续后续清理。

截图容错策略：

- 截图前先等待 800 毫秒，让完成页、字体和异步资源稳定。
- 首次使用 full-page 截图，超时时间 15 秒。
- full-page 截图失败后再等待 1200 毫秒，降级为 viewport 截图。
- 两次截图都失败时，写入一个有效的 1x1 PNG 占位图，日志记录 `Screenshot failed for <suffix>`。
- 调用处使用 `runCatching` 包住 `filled` 和 `submitted` 截图，确保截图异常不影响填写和提交主流程。

导航容错策略：

- 首次进入问卷 URL 时使用 `WaitUntilState.DOMCONTENTLOADED`。
- 每次导航超时 30 秒。
- 最多尝试 2 次。
- 第 1 次失败后等待 2 秒再重试。
- 2 次都失败时抛出 `Navigate failed after retries.`，该行按失败处理。
- 导航完成后的 `NETWORKIDLE` 等待只是辅助稳定页面，超时上限为 8 秒，失败只记 debug，不影响主流程。

## 常见排障

- 如果停在抽奖页，先看日志是否出现 `submit completion page detected`。出现该日志说明抽奖页已被识别为完成页，后续应进入下一行。
- 如果日志停在 `fillRow completed successfully` 但没有 `Row N succeeded on attempt`，优先检查 page/context 释放阶段。
- 如果出现 `submit completion signal was not detected`，说明提交后 12 秒内未匹配完成页规则，可能是安全验证、网络慢、二次确认或新的完成页文案。
- 如果出现 `Timeout 30000ms exceeded` 且日志里有 `Navigate attempt 1 failed`，说明首跳页面仍然超过 30 秒；如果两次都失败，再检查代理速度、问卷星可用性或是否需要继续提高导航超时。
- 如果出现 IP 记录失败，只影响 IP 写回，不代表问卷提交失败。
- 如果问卷星出现安全验证，优先检查验证码配置、填写耗时、来源 UA、代理质量和提交频率。
- 如果截图只有 `row-N-filled.png` 没有 `row-N-submitted.png`，说明提交分支没有完整走到提交后截图，检查验证码处理和点击提交是否成功。
- 如果日志出现 `Protocol error (Page.captureScreenshot)`，现在应只记录 warn 并生成占位 PNG；如果整行仍失败，说明失败点不在截图，而在截图之前的填写、验证码或提交逻辑。
- 如果 Chrome/Edge 启动失败，检查 `QuestionnaireRunner.launchBrowser` 中枚举的本机浏览器路径。
- 如果 Gradle 编译失败，先确认 JDK 21 可用，并使用项目自带 `gradlew.bat`。

## 下次修改优先入口

- 改提交后识别、截图、关闭逻辑：优先看 `QuestionnaireRunner.kt`。

- 改浏览器 UA、移动端、微信端行为：优先看 `automation/browser/ContextInitializer.kt`。
- 改代理池、换 IP：优先看 `automation/proxy` 和 `config/automation-profiles.json`。
- 改 Excel 读取、自动映射、IP 写回：优先看 `ExcelService.kt`。
- 改前端表单、任务轮询、日志展示：优先看 `src/main/resources/static/app.js`。
- 改 API 或静态资源入口：优先看 `Application.kt`。
